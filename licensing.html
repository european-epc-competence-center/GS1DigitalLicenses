Digital Licensing

GS1 Licensing provides the root of trust for the globally unique numbering systems GS1 Digital Licenses are digital representations of the existing GS1 Licenses. These Digital license can be verified by machines in B2B transaction. Why might digital verification of GS1 licecnses be beneficial:

  • To prove a GS1 MO is authentic within the GS1 system for issuing Licenses
  • To prove a MC or MO is authorized to create specific GS1 ID Keys
  • To establish provenance over GS1 ID Keys created by MC or MO
  • To establish provenance over data about GS1 ID Keys created by MC or MO

Digital Licensing in a Nutshell

GS1 Digital Licenses parallel the licenses discussed here.

A Digital GS1 Prefix License is issue by GO to a specific MO. It authorizes them to issue GS1 Company Prefixes (GCPs singular GCP) and single issue identiiers from a specific GS1 prefix value. GS1 Prefix licenses are typically 3 digit numbers, with exceptions listed below. For example, GS1 New Zealand is issued prefixes 940 - 949.

A Digital GCP is isssued from and MO to a MC to authorize them to issue GS1 ID Keys from a specific number range. GCPs have different lengths depending on the needs of the MC. GCPs range from 6 digits to 11 digits. They always begin with the prefix value. The fewer digits a GCP has, the more individual ID keys it can create. For example, a 10 digit GCP (e.g. 08600058400) can only create 10 ID keys of each type while a 6 digit GCP can create many thousands of ID keys.

A Digital Single ID Key License authorizes a MC to use a specific GS1 ID Key for their business operations. Single Issue ID Key Licenses are often used by small business to generate a few GTIN numbers for their products. They are also used to issue GLN numbers to identify organizations in the global supply chain that participate, but are not creating product, shipments or assets.

Digital License Credentials Overview

GS1 Digital Licenses use the W3C Verifiable Credentials V2.0 Format. This credential format uses JSON-LD to encapsulate the details of a license in the format that is well understood worldwide. In addition to the W3C VC JSON-LD context, all GS1 License Credentials must also contain the GS1 License Context.

licenseValue

Every license credential includes a license value. The license credential type is usually sufficient to define, together with the license value, the “space” or “range” of the GS1 identification system granted to the licensee. The exception to this is any license credential that has to do with a GS1 identification key, which requires an identification key type as an additional qualifier. For license credentials that have to do with prefixes, the license value is in normalized GS1 (i.e., non-U.P.C.) form. Where the license credential is within the U.P.C. range, the alternative license value holds the U.P.C. form. For license credentials that have to do with GS1 identification keys, the license value is the key, including the check digit or check characters if applicable, and excluding the serial component if applicable. When the license is specifically for a GTIN, the license value is the GTIN in its minimum format: GTIN-8, GTIN-12, GTIN-13 (which must not start with a zero), or GTIN-14 (which must start with indicator digit 9). Where the license credential is for a GTIN-8, GTIN-12, or GTIN-13, the alternative license value is the GTIN normalized to 13 digits (by padding with zeros on the left) and with the check digit removed (resulting in a 12-digit string); this value is used when verifying a GTIN with indicator digit 1-8 against a license.

alternativeLicenseValue

Alternative value of the license. For most licenses, the alternative value of the license is the bridge between the U.P.C. system and the rest of the GS1 system: if the license value starts with zero, then the alternative license value is the same as the license value but without the starting zero.
